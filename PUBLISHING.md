# Preparing a version of NZ Base for Publication with the IG Publisher using `-go-publish`

## Overview

The NZBase IG has previously used a manual process to prepare it for publishing before the functionality / documentation using the IG publisher for preparing a versioned publication was available. The HL7 FHIR IG Publisher now supports a `-go-publish` flag that automates the versioned publication workflow. Once configured and run, the publisher:

- generates a versioned snapshot directory (in a designated webroot directory) that mirrors the intended web server layout
- maintains a `package-list.json` version history
- generates a `history.html` page and a machine-readable `history.json`
- produces redirect configuration for supported web servers
- updates a package feed (`package-feed.xml` / `publication-feed.xml`) file that the HL7 FHIR package registry polls to discover new releases. 
- rewrites the IG's own location URLs and internal HTML hyperlinks in the published output so they point at the versioned publication path e.g. `https://fhir.org.nz/ig/base/3.1.0/...` (artifact canonicals like `StructureDefinition.url` are stable identifiers and are not changed)
- runs publication-specific QA checks that are stricter than a regular build (e.g. blocks publish if there are unresolved errors, broken links, missing dependencies) and updates a `qa.html` publication QA report for the published version

This output can then be deployed to the web server.. This documentation will provide a step-by-step guide for preparing and executing a formal publication.

## Setup

### Directory setup

Create a single workspace directory that will hold both the mirror of the publication website (the **webroot**) and the cloned source repository for NZ Base. The IG Publisher reads from the IG source and writes the published output into the webroot, so they need to coexist local to each other.

```
publication/                          ← workspace root (any name/location you choose)
  webroot/                                ← local mirror of fhir.org.nz
    ig/
      base/                               ← IG dir (matches published URL path i.e. fhir.org.nz/ig/base)
        package-list.json                 ← bootstrap once, then maintained by publisher
        history.html, history.json        ← generated by publisher
        web.config / .htaccess            ← generated redirect config
        package-feed.xml                  ← generated package feed
        3.1.0/                            ← versioned snapshot created by go-publish
        3.0.1/, 3.0.0/, ...               ← prior versions accumulate here over time
  sources/                                ← cloned IG source repositories
    nzbase/                               ← clone of https://github.com/HL7NZ/nzbase
  ig-history/                             ← clone of HL7/fhir-ig-history-template
  fhir-web-templates/                     ← clone of HL7/fhir-web-templates
  publisher.jar                           ← IG Publisher - use latest version
```

Notes:
- The workspace root name and location are arbitrary, but the **internal structure of `webroot/` must mirror the live website exactly** — the publisher uses paths relative to the destination to locate prior versions and write the new one.
- The IG has its own subdirectory under `webroot/ig/`, matching its published URL path (`ig/base` for NZ Base).
- Source repositories live separately from the webroot so they can be cloned, pulled, etc. independently of the publication staging area. If you want to manage the publication directory using git, this can be done by creating a repo for `publication/` and adding the `sources/nzbase/` to `.gitignore` (so the actual IG source is managed as usual when editing it)
- `publisher.jar` should be the latest version of the publisher and can be placed at the workspace root or alternatively, the version in `input-cache/publisher.jar` can be used.

#### Cloning NZ Base source

Clone the IG you intend to publish into the `sources/` directory. For NZ Base:

```bash
cd publication/sources
git clone https://github.com/HL7NZ/nzbase.git
```

Source repositories are only read by the publisher — go-publish does not modify them.

---
#### Cloning the shared templates

The IG Publisher uses two shared template repositories during publication. Clone them at the workspace root:

```bash
cd publication
git clone https://github.com/HL7/fhir-ig-history-template.git ig-history
git clone https://github.com/HL7/fhir-web-templates.git fhir-web-templates
```

- **`ig-history`** (cloned from `fhir-ig-history-template`) — the template the publisher uses to render `history.html` (the page that lists all published versions of an IG).
- **`fhir-web-templates`** — shared HTML/CSS templates that the publisher uses to render the web shell around the published IG (the site chrome, root-level pages). This is separate from any per-IG template such as `nzBaseTemplate`.

Both repositories should be kept up to date with `git pull` before each publication run.

### Required config files

The publication workflow touches several config files across the IG source repo, the webroot, and the workspace. Grouped by when each needs attention:

#### Per-release updates (IG source repo)

Files inside the IG source repo (`sources/nzbase/`) that need updating before each publication run.

| File | Location | What to update |
|---|---|---|
| `sushi-config.yaml` | repo root | Bump `version` to the release version; change `releaseLabel` from `Draft` to `Release` (and back to `Draft` after publication when bumping to the next dev version) |
| `publication-request.json` | repo root | Set `version`, `path` (versioned URL), `desc`, `descMD`, `status`, `sequence`, `milestone`, and `first` (only `true` for the very first ever publication of this IG). Created fresh per release |
| `ig.ini` | repo root | Stable — no per-release changes required |



##### `publication-request.json` (IG repo root — created per release)
Example for 3.1.0:
```json
{
  "package-id": "fhir.org.nz.ig.base",
  "version": "3.1.0",
  "path": "https://fhir.org.nz/ig/base/3.1.0",
  "milestone": true,
  "status": "release",
  "sequence": "Publications",
  "desc": "Release 3.1.0 of the NZ Base Implementation Guide",
  "descMD": "Release 3.1.0 of the NZ Base Implementation Guide",
  "first": false,
  "title": "HL7® FHIR® New Zealand Base Implementation Guide",
  "changes": "history.html"
}
```
Note: `first: true` only for the very first ever publication. `status` options: `release`, `trial-use`, `ballot`, `draft`.

#### One-time setup (webroot)

Files inside the webroot that need to be created manually the first time, then the publisher maintains them.

| File | Location | What to update |
|---|---|---|
| `package-list.json` | `webroot/ig/base/package-list.json` | Bootstrap once with the IG's prior published version history (see [Bootstrapping package-list.json](#bootstrapping-package-listjson-for-an-ig-with-prior-releases)). After the first go-publish run, the publisher updates this file automatically |

#### Shared template repositories (workspace root)

Cloned once, then kept current with `git pull` before each publication. No editing required.

| Repo | Location | What to update |
|---|---|---|
| `ig-history` | `<workspace>/ig-history/` | `git pull` — no edits |
| `fhir-web-templates` | `<workspace>/fhir-web-templates/` | `git pull` — no edits |

#### External (out of repo)

Files maintained outside this workspace but that the publication relies on.

| Resource | Location | What to update |
|---|---|---|
| HL7NZ package feed | `https://github.com/HL7NZ/hl7-nz-package-feed/blob/main/package-feed.xml` | After publication, update with the new version entry produced by go-publish so the HL7 FHIR package registry can discover the release.


### Pre-publication checklist
Steps to complete in the IG source before running go-publish:
- Fetch latest terminology from NZHTS (`node scripts/fetch-nzhts-resources.js`) - NZHTS authoring environment if there are changes made that haven't been syndicated to prod yet
- Confirm the version number in `sushi-config.yaml` is correct
- Change `releaseLabel` from `Draft` to `Release` in `sushi-config.yaml`
- Do a full clean build (`_genonce.sh` / `_build.bat build`) and verify no errors
- Create/update `publication-request.json` in the repo root (see below)

### Running go-publish

Command (Linux/macOS):
```bash
java -jar input-cache/publisher.jar \
  -go-publish \
  -source . \
  -destination /path/to/webroot
```

Command (Windows):
```bat
java -jar input-cache\publisher.jar -go-publish -source . -destination C:\path\to\webroot
```

Optional flag: `-no-gen` — skips regenerating the IG and uses the existing `output/` directory. Useful if you just ran a clean build.

After the run, the publisher will have:
- Created `webroot/ig/base/3.1.0/` with the full IG content
- Updated (or created) `webroot/ig/base/package-list.json`
- Created `webroot/ig/base/history.html`
- Created redirect configuration in `webroot/ig/base/`


### Redirect configuration and URLs

1stdomains looks like it uses Apache (with an nginx reverse proxy in front), so redirect rules used will be the generated `.htaccess` files using `mod_rewrite`.

#### Auto-generated publication redirects (`.htaccess`)

When go-publish runs, it produces an `.htaccess` file inside `<webroot>/ig/base/` containing `mod_rewrite` rules that redirect requests for the IG's base path (`/ig/base/`) and the IG resource canonical (`/ig/base/ImplementationGuide/fhir.org.nz.ig.base`) to the current published version.

Apache picks this up automatically once the file is deployed alongside the IG content — no manual editing required. Each go-publish run regenerates the file so the redirect always points at the newly published version.

#### Canonical URLs in use

The NZ Base IG uses three different URL families for the canonical URLs it defines for artifacts in the IG. These are seperate to the URL the IG is hosted at (`https://fhir.org.nz/ig/base`) - although the IG resource itself is also using this URL as its canonical URL and this is the canonical defined in `sushi-config.yaml` (at least during the 3.0.1 build). Each of the canonical families has its own resolution behaviour:

| Artifact type | Canonical URL pattern | Example |
|---|---|---|
| The IG resource itself | `http://fhir.org.nz/ig/base/...` | `http://fhir.org.nz/ig/base/ImplementationGuide/fhir.org.nz.ig.base` (the only canoincal using this domain) |
| Profiles, extensions, other StructureDefinitions | `http://hl7.org.nz/fhir/...` | `http://hl7.org.nz/fhir/StructureDefinition/NzPatient` |
| ValueSets | `https://nzhts.digital.health.nz/fhir/ValueSet/...` | `https://nzhts.digital.health.nz/fhir/ValueSet/coverage-category-code` |
| NamingSystems and CodeSystems | `https://standards.digital.health.nz/ns/...` | `https://standards.digital.health.nz/ns/dental-council-id` |

Resolution:
- `hl7.org.nz/fhir/...` URLs are under HL7NZ's control and currently resolve to the Terminz FHIR server that returns the JSON representation of the resource referenced by the canonical. These canonicals could optionally be redirected to the published IG pages on fhir.org.nz so that resolving a canonical brings up the human-readable profile page (or the raw JSON), served from the IG/webserver rather than a separate FHIR server.
- `nzhts.digital.health.nz` and `standards.digital.health.nz` are HNZ managed domains. Currently these don't resolve to anything. The ValueSets URLs could be made to resolve by ensuring the ValueSet resource id on NZHTS matches the path (which would then be served by OntoServer). We could also investigate if we were able to redirect the standards.digital.health.nz URL requests.
